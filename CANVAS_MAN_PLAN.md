# Canvas Man Website – Joomla to Static Site Conversion Plan

## Overview

Convert the Joomla-driven website at https://www.canvasman.com into a fully static website built with **Eleventy (11ty)**. The local `site_files/` folder contains the original Joomla install (tarball) and database. The goal: a static portfolio site that's easy to update — business info, pages, and gallery images all editable from simple files in git.

## Stack & hosting

- **Generator:** Eleventy 3.x
- **Image processing:** `@11ty/eleventy-img` (responsive resizing at build time)
- **Templating:** Nunjucks
- **Styling:** Hand-written CSS (no framework)
- **Source of truth for content:** Markdown + a single YAML data file
- **Hosting:** Plain web server (static files), source kept in git, no runtime needed

## Build phases

1. **Phase 1 — Extract & organize assets** ✅
   - Parse FW Gallery DB to map image filenames → categories
   - Extract only original-resolution images from tar
   - Place them under `src/images/gallery/<area>/<category>/`

2. **Phase 2 — Eleventy scaffolding** ✅ (in progress)
   - `package.json`, `.eleventy.js`, `.gitignore`
   - Add `@11ty/eleventy-img` for responsive images

3. **Phase 3 — Layouts & styling**
   - `base.njk`, header, footer, nav partials
   - Clean modern responsive CSS

4. **Phase 4 — Content**
   - `_data/site.yml` with all editable business info
   - Markdown pages for Home, About, Contact, Residential, Commercial, Marine
   - Pull descriptions from DB (`jos_fwg_projects.description`)

5. **Phase 5 — Gallery system**
   - Eleventy data file walks `images/gallery/` and emits one page per category
   - Lightbox JS for image viewing
   - Responsive thumbnails generated by `eleventy-img`

6. **Phase 6 — Build, verify, deploy**
   - `npm install && npm run build`
   - Smoke-test in browser
   - Output: `_site/` directory ready to rsync to any web host

## How content updates work post-build

| To change... | You edit... |
|---|---|
| Address, phone, hours, email | `src/_data/site.yml` |
| Page text | `src/<page>.md` |
| Add gallery photos | Drop into `src/images/gallery/<area>/<category>/` |
| Add new gallery category | Make a new folder under `src/images/gallery/<area>/` |
| Add a new page | Create `src/<name>.md` and add to nav in `site.yml` |

## Image curation backlog

Deferred curation passes — do these before publishing.

- [x] **Image variant upgrade** — initial extraction grabbed display-sized variants (suffixed `-2`, `-3`, etc.) for 38 images. Replaced each with the highest-resolution version available in the original tar.
- [x] **Exact duplicate removal** — removed 16 same-category byte-identical dupes. 4 cross-category dupes intentionally preserved (one photo legitimately illustrates two categories).
- [x] **Low-resolution archive** — moved 24 images with min-dimension < 500px to `archive/low-resolution/` (preserving directory structure) with a documentation log at `archive/ARCHIVE_LOG.md`. Files are not deleted; they can be restored by moving back into `src/images/gallery/`. **375 images now live.**
- [x] **Empty-category report (automatic)** — the gallery data loader detects category folders with no images and writes `archive/EMPTY_CATEGORIES.md` on every build. Empty categories are silently skipped from the site (no page, no listing) so they don't show as broken pages. The report lists each empty folder and notes whether any of its images were archived, so it's easy to restore or remove.
- [x] **Perceptual near-duplicate detection** — `tools/find-similar.js` uses sharp-phash (64-bit pHash) + union-find to group similar images at a configurable Hamming-distance threshold (default 8). Output goes to `build/similar-groups.json`. The `/review/similar/` page reuses the contact-sheet review JS for click-to-cut → Export → `apply-cuts`. Cross-category and byte-identical groups are flagged with badges. See README "Curating gallery photos" for the workflow. **First run uncovered 17 SHA256-identical files (8 of them in `residential/pool-cabana-covers/`) that the original exact-dedup pass missed — likely introduced when the image-variant upgrade later replaced display variants with the same source file. Follow-up PR will use the new tool to clean those up.**
- [x] **Manual contact-sheet review** — built as a dev-only tool. `npm run review` boots the dev server at port 8081 with `REVIEW_MODE=1`, exposing `/review/` (index) and `/review/<area>/<category>/` (per-category grids). Click any photo to mark it for cut; selections persist in `localStorage` across pages; sticky bottom bar shows count + Export/Clear. `npm run apply-cuts -- cuts.json` moves marked files to `archive/manual-curated/` (preserving structure) and appends a dated entry to `archive/ARCHIVE_LOG.md`. Review pages are excluded from production builds — never appear on `preview.canvasman.com`. See README "Curating gallery photos" for the full workflow.

## Deployment

- [x] **Preview hosting** — site is built and deployed by `.github/workflows/deploy.yml` on every push to `main`. Custom domain `preview.canvasman.com` is wired via a CNAME record at the registrar and the `src/CNAME` file in the repo.
- [x] **Business info confirmed** — name, address, phone, emails, hours all confirmed against the live site and stored in `src/_data/site.yml`. Tagline and copy adapted from the live canvasman.com.
- [x] **Uptime monitoring** — `https://preview.canvasman.com/` is monitored by [UptimeRobot](https://uptimerobot.com) (free tier, 5-minute interval, multi-region checks). Email alerts go to the account owner; the monitor runs on infrastructure independent of GitHub so an outage on either side is still visible. When canvasman.com itself is cut over to this site, add a second monitor for the production URL — keep both so reliability can be compared. The dashboard is at https://uptimerobot.com/dashboard. Account-level changes (alert contacts, status page, monitor settings) happen there, not in this repo.
- [x] **Production hosting choice** — staying on GitHub Pages. Free, sufficient for the bandwidth this site will see, and the existing deploy / preview / TLS setup already works. Revisit if (a) bandwidth approaches the 100 GB/mo soft cap, (b) we need server-side form handling or edge functions, or (c) the contact form (item below) pushes us toward Netlify for its built-in forms.
- [ ] **Cutover window & parallel-run plan** — when we're ready to replace canvasman.com itself, decide the cutover window and whether to keep the legacy Joomla server running in parallel for a grace period (recommend: yes, ~30 days, so any missed redirects can be observed and patched).
- [ ] **Contact form** — currently mailto-only. Decide whether to add a real form (Formspree, Basin, etc. — needs free account + tiny JS) or stay with mailto + tel.
- [ ] **SEO & 301 redirects** — old Joomla site has ~7,000 URL aliases in `jos_sh404sef_aliases`. If/when canvasman.com is replaced, we should preserve the most-trafficked old URLs with 301s to the new equivalents. Probably only the top 20–50 URLs matter; the rest can `410 Gone`.
- [x] **HTTPS / SSL on preview.canvasman.com** — Let's Encrypt certificate auto-provisioned by GitHub Pages once the CNAME record resolved. "Enforce HTTPS" enabled in repo Settings → Pages so HTTP requests redirect to HTTPS.
